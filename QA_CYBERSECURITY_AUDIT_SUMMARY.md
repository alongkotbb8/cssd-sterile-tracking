# เอกสารสรุปผลการตรวจสอบระบบ CSSD

วันที่ตรวจสอบ: 18 กรกฎาคม 2026  
ขอบเขต: QA, UX/UI, Cybersecurity และความพร้อมใช้งาน  
วิธีตรวจสอบ: ตรวจสอบโค้ดแบบ Read-only

## 1. บทสรุปผู้บริหาร

ระบบมีโครงสร้างหลัก ได้แก่ การสร้างห่อ การออกเลขรัน การสแกนนำเข้า–เบิกออก–ส่งคืน การพิมพ์ Label ผ่านระบบพิมพ์และ Bluetooth รายงาน PDF การแจ้งเตือน และ Audit Log

อย่างไรก็ตาม ระบบยังไม่ควรนำไปใช้งานจริงในกระบวนการ CSSD เนื่องจากพบข้อบกพร่องระดับ Critical ที่อาจกระทบความปลอดภัยผู้ป่วย ได้แก่:

1. Label ของห่อที่ยังไม่ผ่านการนึ่งสามารถแสดงวันที่นึ่งและวันหมดอายุโดยประมาณ
2. ระบบสามารถนำห่อไปผูกกับรอบนึ่งย้อนหลัง หลังรอบนั้นผ่านการตรวจแล้ว
3. ผู้ใช้บทบาท CSSD สามารถบันทึกผล CI/BI และรับรองรอบนึ่งได้

ผลการประเมินโดยรวม: **ไม่ผ่านเกณฑ์ Production Readiness**

## 2. ข้อบกพร่องที่ตรวจพบ

| ระดับ | ข้อบกพร่อง | ผลกระทบ |
|---|---|---|
| Critical | พิมพ์วันที่นึ่งและวันหมดอายุโดยประมาณก่อนนึ่งจริง | Label อาจสื่อว่าห่อผ่านการฆ่าเชื้อแล้ว |
| Critical | ผูกห่อเข้ากับ Batch ที่ผ่านผลตรวจแล้วได้ | ห่อที่ไม่ได้อยู่ในรอบจริงอาจกลายเป็น `STERILE` |
| Critical | Endpoint บันทึก CI/BI ไม่จำกัดบทบาท | บุคลากรที่ไม่มีอำนาจสามารถรับรองรอบนึ่ง |
| High | ไม่มีระบบ Tagging จริง | ไม่สามารถติดแท็ก ค้นหา หรือจัดกลุ่มตามความต้องการ |
| High | Offline-first ยังไม่ทำงานจริง | สแกนและทำรายการไม่ได้เมื่อเครือข่ายขัดข้อง |
| High | การพิมพ์ไม่บันทึก `printedAt` และ `reprintCount` | ตรวจสอบการพิมพ์ซ้ำและผู้ดำเนินการไม่ได้ |
| High | Audit Log ไม่อยู่ใน transaction เดียวกับรายการหลัก | ข้อมูลเปลี่ยนแล้วแต่ Audit Log อาจไม่ถูกบันทึก |
| High | ความหมายวันหมดอายุยังไม่ชัดเจน | อาจบล็อกห่อตั้งแต่ 00:00 ของวันที่หมดอายุ |
| Medium | ไม่มีรายงาน Excel | ยังไม่ครบตามขอบเขตระบบ |
| Medium | PDF อาจต้องดาวน์โหลดฟอนต์ขณะสร้าง | รายงานอาจสร้างไม่ได้เมื่อ offline |
| Medium | Server URL ไม่บังคับ HTTPS | เสี่ยงส่ง JWT และข้อมูลผ่านเครือข่ายไม่เข้ารหัส |
| Medium | CORS เปิดทุก Origin หากไม่ได้ตั้งค่า | เพิ่มพื้นที่การโจมตีใน Production |
| Medium | Cleanup ลบ Movement และ Audit Log ได้ | สูญเสียประวัติ traceability ระยะยาว |

## 3. ผลการตรวจสอบตามฟังก์ชัน

### 3.1 การพิมพ์ Label

สถานะ: **มีระบบ แต่ยังไม่ปลอดภัยพอ**

รองรับ:

- System Print
- Bluetooth Classic
- BLE
- ภาษาไทยด้วย Bitmap
- QR ที่บรรจุเฉพาะ Package ID
- Label ขนาดประมาณ 60 × 40 มม.

ปัญหาสำคัญ:

- Label ก่อนนึ่งแสดงวันนึ่งสมมติ
- ไม่มีการบันทึกประวัติการพิมพ์และพิมพ์ซ้ำใน backend
- ยังไม่มีหลักฐานการทดสอบกับเครื่อง FlashLabel A318BT จริง
- ไม่สามารถยืนยันจาก Static Review ว่า Label อ่านได้ทุกใบและไม่มีการตัดข้อความ

### 3.2 การสแกนนำเข้า

สถานะ: **ไม่ผ่าน**

ระบบตรวจสถานะ `PACKED` และต้องเลือก Batch ที่ `PASSED` แต่ลำดับกระบวนการผิดหลัก traceability เพราะนำห่อไปผูกกับ Batch หลังผ่านผลตรวจแล้ว

กระบวนการที่ควรเป็น:

`สร้างห่อ → ผูกห่อเข้ารอบ PENDING → เริ่มนึ่ง → บันทึก CI/BI → ผ่านผลตรวจ → เปลี่ยนห่อเป็น STERILE`

### 3.3 การสแกนเบิกออก

สถานะ: **ผ่านบางส่วน**

สิ่งที่ทำถูกต้อง:

- บังคับเลือกแผนกปลายทาง
- ชื่อผู้รับไม่บังคับ
- บล็อกห่อหมดอายุ
- ป้องกันการสแกนซ้ำพร้อมกัน
- รองรับ `PACKED_OUT`

สิ่งที่ควรเพิ่ม:

- ยืนยันซ้ำเป็นพิเศษเมื่อเบิก `PACKED_OUT`
- จำกัดปลายทางของ `PACKED_OUT` ให้เป็น Department ชนิด `external`
- เพิ่มเสียงและภาพแจ้งเตือนที่ชัดเจนสำหรับของหมดอายุ
- เพิ่ม Idempotency Key สำหรับคำขอจาก offline sync

### 3.4 การส่งคืน

สถานะ: **ผ่านบางส่วน**

ระบบบันทึกแผนกที่ส่งคืนและรองรับ:

- `ISSUED → RETURNED`
- `PACKED_OUT → PACKED`

ควรเพิ่มการตรวจว่าแผนกผู้ส่งคืนตรงกับแผนกที่เบิกออกล่าสุดหรือไม่ หากไม่ตรงควรเตือนและบันทึกเหตุผล

### 3.5 รายงาน

สถานะ: **ยังไม่ครบ**

มี:

- รายงานตามช่วงเวลา
- สรุป IN, OUT และ RETURN
- PDF ภาษาไทย
- รายการของที่ยังไม่ส่งคืน
- Dashboard สรุปสถานะ

ยังขาด:

- Excel export
- รายงานอัตราการส่งคืนแยกตามแผนก
- รายงาน Recall
- รายงานประวัติพิมพ์ Label
- รายงานเหตุการณ์ผิดปกติและรายการถูกบล็อก
- Custom date range และตัวกรองที่ละเอียด
- Digital signature หรือหมายเลขเวอร์ชันรายงาน

### 3.6 Tagging

สถานะ: **ยังไม่มีฟังก์ชัน**

ควรกำหนดก่อนว่า Tagging หมายถึง:

- Tag ประเภทชุด
- Tag ความเร่งด่วน
- Tag สถานที่จัดเก็บ
- Tag ความเสี่ยง
- Tag สำหรับค้นหาและรายงาน
- RFID/NFC Tag

หากเป็น Tag ทางซอฟต์แวร์ ควรมีโมเดล `Tag` และความสัมพันธ์กับ Package หรือ SetTemplate พร้อมสิทธิ์จัดการ Tag

## 4. การประเมิน UX/UI

จุดที่ทำได้ดี:

- UI ภาษาไทย
- แยกโหมดนำเข้า เบิกออก และส่งคืนชัดเจน
- บังคับเลือกปลายทางก่อนยืนยัน
- มีหน้าต่างยืนยันก่อนบันทึก
- แสดงเหตุผลเมื่อรายการสแกนไม่ผ่าน
- มีแนวทางแก้ปัญหาสิทธิ์กล้องและ Bluetooth
- รองรับการสแกนหลายห่อในหนึ่งรายการ

สิ่งที่ควรปรับปรุง:

- ห้ามแสดงปุ่มพิมพ์ Label ที่มีวันนึ่งก่อนมีข้อมูลจริง
- เพิ่มหน้าสรุปผลหลังทำรายการ แทนการแจ้งด้วย SnackBar เพียงอย่างเดียว
- เพิ่มปุ่ม “ลองใหม่เฉพาะรายการล้มเหลว”
- เพิ่มตัวนับรายการสำเร็จ/ล้มเหลวที่เด่นชัด
- เพิ่มเสียงต่างกันระหว่างสำเร็จ คำเตือน และถูกบล็อก
- แสดงสถานะเชื่อมต่อเครื่องพิมพ์ตลอดเวลา
- เพิ่ม Test Print และ Print Preview
- ป้องกันการกดล้างรายการโดยไม่มีการยืนยัน
- รองรับตัวอักษรขนาดใหญ่และการใช้งานด้วยถุงมือ
- เพิ่ม Manual Entry เมื่อกล้องเสีย โดยจำกัดสิทธิ์และบันทึก Audit Log

## 5. การประเมิน Cybersecurity

สิ่งที่ทำได้ดี:

- เก็บ JWT ใน Secure Storage
- JWT มีอายุใช้งาน
- ตรวจสถานะผู้ใช้และบทบาทใหม่ทุก request
- ไม่มี JWT secret สำรองในโค้ด
- มี Validation Pipe
- ป้องกัน brute-force ขั้นพื้นฐาน
- ซ่อนรายละเอียด internal error
- ตรวจสิทธิ์สำหรับ Master Data และ Recall บางส่วน

สิ่งที่ต้องแก้:

1. จำกัดสิทธิ์บันทึกผล CI/BI
2. บังคับ HTTPS สำหรับ Production
3. จำกัด CORS ด้วย allowlist
4. ปิดหรือจำกัด Swagger ใน Production
5. ทำ rate limiting ผ่าน shared store เมื่อมีหลาย API instance
6. เพิ่ม account lockout หรือ progressive delay
7. ทำ Audit Log แบบ append-only และป้องกันการแก้ไข
8. ห้ามลบ Audit Log โดยตรง
9. เพิ่ม security headers และ request-size limit
10. เพิ่มการตรวจ dependency vulnerability เป็นประจำ
11. เพิ่ม certificate pinning หากนโยบายโรงพยาบาลกำหนด
12. เพิ่ม session/device management และ remote logout

## 6. ลำดับการแก้ไขที่แนะนำ

### ก่อนทดสอบระบบอีกครั้ง

1. แก้ลำดับ Sterilization Batch
2. ห้ามพิมพ์วันนึ่งสมมติ
3. จำกัดสิทธิ์บันทึก CI/BI
4. ทำ Audit Log ให้อยู่ใน transaction
5. บันทึกประวัติพิมพ์และพิมพ์ซ้ำ
6. กำหนดความหมายวันหมดอายุให้ชัดเจน

### ก่อน Pilot

1. ทดสอบเครื่องพิมพ์ A318BT จริง
2. ทดสอบ Label อย่างน้อย 100 ใบต่อเนื่อง
3. ทดสอบ QR หลังพิมพ์ด้วยโทรศัพท์หลายรุ่น
4. เพิ่ม Excel และรายงาน Recall
5. เพิ่ม Tagging ตามนิยามที่ยืนยัน
6. ทำ offline queue และ conflict handling
7. ทำ Security API testing และ RBAC matrix

### ก่อน Production

1. Penetration test
2. Load และ concurrency test
3. Backup/restore drill
4. Recall drill
5. Disaster recovery test
6. User Acceptance Test กับเจ้าหน้าที่ CSSD
7. จัดทำ SOP และคู่มือฉุกเฉินเมื่อระบบหรือเครื่องพิมพ์ล่ม

## 7. เกณฑ์อนุมัติใช้งานจริง

ระบบควรได้รับอนุมัติเมื่อ:

- ไม่มี Critical หรือ High defect ค้างอยู่
- Label ทุกใบใช้ข้อมูลจาก backend ที่ผ่านการรับรองแล้ว
- QR อ่านสำเร็จตามเกณฑ์อย่างน้อย 99.5%
- การสแกนซ้ำไม่ทำให้เกิด Movement ซ้ำ
- Recall แสดงตำแหน่งห่อครบทุกใบ
- ระบบบล็อกของหมดอายุได้ 100%
- RBAC ผ่านทุกกรณี
- Audit Log ครบและไม่สามารถแก้ไขย้อนหลัง
- Offline sync ไม่สร้างเลขหรือ Movement ซ้ำ
- เครื่องพิมพ์จริงผ่านการทดสอบต่อเนื่อง
- ผ่าน UAT และการอนุมัติจากผู้รับผิดชอบ CSSD

## 8. ข้อสรุป

ระบบมีพื้นฐานที่ดีและหลายส่วนออกแบบตรงตามกฎโดเมน แต่ข้อผิดพลาดเรื่อง Label และลำดับรอบนึ่งมีความเสี่ยงสูงต่อ traceability และความปลอดภัยผู้ป่วย จึงควรหยุดการนำไปใช้งานจริงไว้ก่อน จนกว่าข้อบกพร่องระดับ Critical และ High จะได้รับการแก้ไขและทดสอบซ้ำครบถ้วน

---

เอกสารนี้เป็นผลจากการตรวจสอบโค้ดแบบ Static Review ยังไม่ได้ยืนยันผลกับอุปกรณ์เครื่องพิมพ์จริง สภาพเครือข่ายจริง หรือฐานข้อมูล Production
